# Isolate the deferred scheduling integration

The first implementation milestone will make onboarding, LinkedIn reply extraction and generation, manual rewriting, idea discovery, post generation, history, settings, and provider-key handling functional end to end. Scheduling will be represented only by its approved Schedule Preview and will not create or persist a schedule, call a mock service, send email, or produce Scheduled Idea Results until the separate service API is supplied. Its controls remain interactive for UI testing and Save validates visible fields, then explains that scheduling becomes available when the service is connected while retaining entered values only until the panel closes; no active or next-run status is shown. The UI will depend on typed scheduling contracts so signup and the real API can be integrated without rewriting the approved screen.
