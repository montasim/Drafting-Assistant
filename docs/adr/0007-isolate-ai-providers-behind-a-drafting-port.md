# Isolate AI providers behind capability ports

Every AI workflow—including reply analysis, reply generation, rewriting, idea discovery, standalone post drafting, profile derivation, and style analysis—will depend on an application-owned capability interface rather than Gemini- or Groq-specific clients. Provider Readiness requires both credentials to be present and successfully validated before onboarding can finish or any Foreground AI Job can start; Thoughtline will not offer Gemini-only, Groq-only, or unverified degraded operation. Gemini and Groq API origins are optional host permissions requested together, with a plain explanation, during the Connections setup step rather than at extension installation; declining or later revoking either permission leaves setup or Provider Readiness blocked until access is granted again. **Continue** during onboarding and **Check connections** in Settings validate each credential by asking the exact configured Gemini and Groq model to transform fixed, non-user test content into a minimal structured response, then validating that response locally with Zod. Format checks or model-list access alone are insufficient. Each successfully validated credential is encrypted and saved independently even if the other credential fails; a failed candidate never overwrites a previously valid stored credential, and the UI identifies each provider as passed, failed, or unchanged without exposing its response or key. Provider Readiness remains blocked until both stored credentials are valid. Readiness changes never navigate onboarding automatically: the user explicitly presses **Continue**, which validates any changed or unvalidated values and proceeds only if both providers are ready after that user-initiated action. Failed values remain available for correction during the current panel session, and validation never transmits LinkedIn, profile, draft, or History content. Connection tests are event-driven rather than hidden startup or per-job calls: they run during setup, whenever a credential is replaced, or when the user invokes **Check connections**. Between those events, previously validated providers remain ready unless a real request proves an authentication or model-access failure, at which point the affected provider is marked invalid; the current Gemini job may still complete through approved Groq fallback, but subsequent AI jobs are blocked until both providers validate again. Network, timeout, rate-limit, and transient service failures do not invalidate a credential. Gemini is always the primary provider and Groq receives one automatic retry after a Gemini timeout, network failure, rate limit, server error, invalid credential, unavailable model, malformed response, or incomplete result. Cancellation, invalid input, unsupported LinkedIn layouts, locally detected context overflow, and provider-independent safety failures do not trigger fallback. Provider adapters own transport and response translation, while orchestration, validation, fallback policy, and domain results remain provider-neutral so future providers can be added or selected without changing UI or use-case code.
